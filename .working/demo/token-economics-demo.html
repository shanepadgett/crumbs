Editing without re-buying the file

Grounded patches: edit by anchor, not by repetition

Standard diff formats make the model retype the old lines it wants to delete or replace. If the model already proved it has the current file (a content-hash base tag), line anchors are enough.

Standard patch — delete 16 lines

*** Update File: src/checkout.ts
@@ class CheckoutService
-  private legacyRetry(attempt: number) {
-    const delay = attempt * 1000;
-    if (attempt > MAX_RETRIES) {
-      this.logger.warn("max retries");
-      return false;
-    }
    … 10 more re-typed lines …
-  }
≈ 230 tokens of output — re-typing code that already exists, at output-token prices (5–6× input price)

Grounded patch — same edit

*** Update File: src/checkout.ts
*** Base: A1B2C3D9   ← hash of the file view the model saw
delete 80..95
insert after 120:
+export const retryEnabled = true;
≈ 30 tokens — and the old read can now be purged from context, because the edit is anchored to the hash, not the prose

Safety property: if the file changed since the model last saw it, the base tag mismatches and the patch is rejected before any write — the model is told to re-read. That's strictly safer than today's string-matching edits, which can silently hit the wrong occurrence. Cheaper and harder to corrupt.

Why it compounds: output tokens cost 5–6× input. Every deleted line a model doesn't retype is the most expensive token class you didn't buy.
The obvious objection, priced honestly

"Doesn't rewriting context break prompt caching?" Yes. Do it anyway.

Providers bill cached input at 10% of list price — but only for an unchanged prefix. Editing old context invalidates the cache from that point on. Here's the honest math on when that's worth it.

Turn 1 — nothing cached yet
Turn 2 — prefix cached (10% price)
Compaction turn — prefix rewritten, cache lost
12 turns
Keep 120k bloated context, perfect cache hits
Rewrite to 30k — pay the cache miss, then cheap
Punchline: a cache hit is a 90% discount on tokens you shouldn't be sending at all. The discount on waste is still waste. Shrink first, cache second — the miss pays for itself in 4–5 turns.
It's not only money

Full context windows make agents dumber — then the rework loop bills you twice

Long-context degradation is well documented: reasoning quality drops as windows fill. When a naive harness hits its limit, it auto-compacts with a lossy summary — the agent forgets what it tried, re-reads files, re-runs tests. You pay for the same work twice.

Straight talk

What I'm claiming — and what I'm not

Everything here is a cost model built on published list pricing, not a benchmark. The mechanisms are designed and specced; the eval plan to prove savings don't hurt task success is written. That measurement is the next step.

Honest tradeoffs (already engineered for)

  • Retrieval round-trips cost tokens — surface stays tiny (2 tools), capsules carry enough signal to avoid most fetches.
  • A capsule could hide a detail the model needed — nothing is destroyed, exact bytes always retrievable by id.
  • Context rewrites bust prompt cache — priced on the cache slide; the miss wins in 4–5 turns.
  • Solve-rate is the real metric — eval plan covers stale reads, needle recovery, post-compaction continuation.

Why this won't come from the vendors

  • Model vendors bill per token; their harnesses optimize capability and lock-in, not our spend.
  • The industry is normalizing "token-maxing" — million-dollar monthly burns with no matching delivery gains. Enterprise AI ROI write-ups keep saying the same thing.
  • Context engineering is a real, separable layer — and it's harness-side, which means it's ours to own.
  • Whoever owns the harness owns the cost curve, the telemetry, and the leverage in every vendor negotiation.
