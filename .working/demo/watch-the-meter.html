Watch the meter

Every coding agent we run today has the same expensive habit: before each step, it re-reads everything it has ever seen, and we pay list price for the re-reading. Nobody chose this. It's the default physics of the agent loop, and the vendors charging by the token are in no hurry to change it. So let me show you one small task, played twice, with the meter running. The problem explains itself in about ninety seconds, and the fix is a harness we can own.

First, the part everyone skips

A language model has no memory. Turn forty doesn't remember turn one; the harness re-uploads turns one through thirty-nine, every time, and the meter runs on all of it. The file the agent read at 2pm gets bought again at 2:01, 2:02, 2:03, for as long as the session lives. Output tokens get the headlines because the per-token rate is five to six times higher. In agent loops the volume runs the other way: almost everything we buy is input, and almost all of that input is something we already bought.

That's the whole setup. Everything else in this note is one question: what should the harness re-send, and what should it stop re-sending?

One failing test, played twice

A checkout test fails on retries. The agent fixes it and the engineer asks for a follow-up. Same model, same prompts, same outcome on both sides. The only difference is the harness. The default harness keeps every byte it ever saw. The token-conscious harness files full outputs in a local artifact store, sends compact summaries with retrieval handles, and marks dead context as dead. Each column shows what gets re-uploaded on the next turn, with a running spend meter. Step through it; the commentary explains each move.

One small task. The default harness ends the afternoon re-sending about 88,000 tokens per turn; the conscious one, about 5,000. Roughly ten times cheaper on the session meter, and nothing was lost: the full logs, the full file, the whole search flood all sit on disk, one retrieval call away. Now notice what the second column was doing. Each of those moves is a mechanism we've designed, and a few already run in my harness today. The next sections show them up close, with the actual payloads, because a bar chart claiming "ours is better" is exactly the kind of slide I don't want to show you.

What the conscious harness actually sends

Reads come back as structure, with the full text on file. For a big file, the model rarely needs all 812 lines to start reasoning. It needs the shape, the ranges that match the task, and a guarantee that exact lines are one call away. The hash matters later: it's what lets the harness prove a cached view is still current, and what anchors edits.

Passing tests are silence. This one already runs in my harness today; I call them quiet validators. The agent never runs the suite at all. Validators run outside the context window after every relevant file change, and they speak only on failure. A green suite costs zero tokens. Compare what each side carries after the same failed run:

Everything bulky collapses to a capsule once it stops being news. Nothing is deleted; the capsule names the artifact, and the model pulls exact bytes back when it needs them. In practice it rarely needs them, because the capsule keeps the part that mattered.

Edits are the worst offender, so here's the actual payload

Our baseline agent edits by quoting: every change repeats the old text and the new text in full, one tool call per change. Those quoted lines are output tokens, the most expensive kind we buy. And because each call is its own turn, three edits to one file re-upload the entire context twice more. The edit itself is cheap. The ceremony around it is not.

Yes, this breaks the cache. Do it anyway.

Anyone who knows the billing model will object here: providers sell cached input at a tenth of list price, and the discount only applies to a byte-identical prefix. Rewriting old context forfeits it for one turn. True, and worth pricing rather than hand-waving.

Break-even lands at four to five turns on current list prices, and sessions run far longer than five turns. A cache hit is a 90% discount on tokens that shouldn't be in the prompt at all. Shrink first. Cache what remains.

The part that isn't about money

Models reason measurably worse as their context fills. Long-context degradation shows up in every published evaluation that looks for it, and our engineers describe it without prompting: the agent "gets dumb" an hour in. Then it gets worse. When a default harness hits its window limit, it auto-compacts: a lossy summary replaces the history. The agent forgets which fixes it already tried, re-reads files it had open, re-runs suites that already passed, and occasionally re-applies a change the engineer reverted on purpose. We pay for the same work twice and the engineer babysits the do-over. This is where "$400 a month in two days" comes from: people aren't burning budget because their agents work too well.

A harness that holds its working set at a quarter of the window never meets that wall. When compaction is genuinely worth doing, it writes a structured dossier: the goal, the user's constraints verbatim, what changed, what failed, and handles to every stored artifact. The next turn recovers detail instead of rediscovering it.

What I'm claiming, and what I'm not

These are modeled costs on published list prices, and every figure states its assumptions. There's no benchmark yet and I won't pretend otherwise. The evaluation plan is written: stale-read traps, needle recovery out of capsuled artifacts, continuation quality after compaction, and a solve-rate regression suite. Solve rate gates everything. Savings that make the agent dumber don't count.

The tradeoffs are engineered, not ignored. Retrieval round-trips cost tokens, so the added tool surface is two tools. A capsule could hide a detail the model needed, so every capsule carries a retrieval handle and the exact bytes stay on disk. Curation breaks the cache, and the break-even math above prices that honestly. Some of this already runs in my harness today; the rest is specced and sequenced.

Here's the existential part, said plainly. Inference is becoming a top-three engineering cost, and the industry's answer so far is rationing: budget caps, usage tiers, engineers waiting for their allowance to reset. Meanwhile the vendors' own harnesses, the ones our engineers use all day, have no incentive to send fewer tokens, because every wasted token is revenue. That layer will not fix itself. It's harness software. We can own it, and whoever owns the harness owns the cost curve, the telemetry, and the leverage in the next vendor negotiation.

Think of it as a hiring decision. The default harness employs someone brilliant, requires them to re-read every email they've ever received before answering each new one, and bills us hourly for the reading. We would fire a person who worked this way. We should fire the harness.

Pay the model to think. Stop paying it to re-read its own inbox.
