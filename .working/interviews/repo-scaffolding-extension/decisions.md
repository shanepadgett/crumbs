# Repo Scaffolding Extension Interview

## Goal

Reach shared design for a Pi extension that scaffolds repo setup deterministically: supported sources, UI flow, version-selection rules, write behavior, and initial scaffold scope. Stop when decisions are enough to build first clean slice without guessing.

## Decisions

- Build Pi extension with slash command for repo scaffolding using built-in Pi UI menus.
- Generate deterministic repo setup across projects where Crumbs is available.
- Use templates to define scaffoldable items/configs and version lookup sources, not pinned default versions.
- Require user to choose versions from presented lists.
- Always write exact pinned versions. Never write `latest`, ranges, fuzzy versions, or mutable channel names.
- UI may show channel labels like `Latest stable (x.y.z)` or `Active LTS (x.y.z)`, but final output writes only exact resolved version.
- Do not provide manual version entry escape hatch.
- Lookup failure blocks affected item and reports clear registry/source error.
- Version sources:
  - mise registry/tools via `mise ls-remote`.
  - npm packages via npm registry metadata.
  - curated GitHub-backed mise tools via GitHub Releases API.
- GitHub support is curated template only, not arbitrary repo input.
- GitHub backend uses releases/tags, not branches.
- No GitHub CLI dependency.
- Do not install dependencies or run tool installs by default. Writing files/tasks is primary behavior; any install/run step needs explicit confirmation later if added.
- Use hybrid scaffold selection: choose one or more profiles, then toggle included scaffold items a la carte.
- Preview step shows concise file/action list with paths only, not full file contents.
- Never overwrite existing files silently.
- For existing unmanaged files, default to skip; replace only with explicit user choice.
- For known structured files like `package.json` and `mise.toml`, merge additive fields/tasks when safe and summarize path/action.
- Do not build complex three-way merge.
- Tool only creates or overwrites/merges selected scaffold outputs. No future update tracking.
- Do not add managed markers or sidecar scaffold state.
- If target file can be handled structurally, use structured merge/update instead of full replacement.
- Structured merge preserves unrelated content and prompts before overwriting selected existing keys.
- Do not include Prettier.
- JS/TS quality stack choice is either Ox stack (`oxlint` + `oxfmt`) or Biome stack (`@biomejs/biome`), not both at same time.
- Markdown linting is independent selectable profile/item and can combine with either Ox or Biome.
- Use `Open Questions` only as parking lot for newly surfaced branches that must be revisited. Do not pre-list the normal next interview question sequence.
- Runtime/package manager selection blends existing repo detection with support for repos that use both Bun and Node.
- Dual Bun/Node repos use primary runner plus secondary runtime: generated tasks use the selected primary runner, while secondary runtime is pinned in `mise.toml` only.
- Existing repo detection preselects runtime/package manager choices and explains detected evidence, but user can override.
- Runtime/package manager detection uses repo metadata files, not installed artifacts.
- Detection evidence includes `package.json#packageManager`, lockfiles, `.npmrc`, and existing `mise.toml` tools.
- Detection precedence: `package.json#packageManager`, then lockfiles, then existing `mise.toml` tools, then ask user.
- JS/TS standalone quality CLIs are managed through mise npm backend with exact pinned versions in `mise.toml`.
- Standalone mise-managed npm CLIs include `oxlint`, `oxfmt`, `@biomejs/biome`, and `markdownlint-cli2`.
- `typescript` is project-coupled compiler/library tooling and is added as exact pinned `package.json` `devDependencies` when TS support is scaffolded.
- Do not create `package.json` solely for standalone lint/format/markdown CLIs.
- If TS support is selected and no `package.json` exists, create minimal `package.json` needed for exact pinned `typescript` devDependency.
- Generated `package.json` stays minimal: dependency manifest only, no scripts, no rich project defaults.
- `mise.toml` owns generated user-facing tasks.
- Generated tasks call mise-provided CLI shims and assume users run `mise install` before task execution.
- Create/update `mise.toml` only when selected scaffold adds mise-managed tools or tasks. Do not create empty/pointless `mise.toml`.
- JS/TS profiles always include runtime pin selection and write exact runtime version to `mise.toml`.
- Profiles include JS/TS, Markdown, YAML, Go, and Swift/Xcodebuild because these are common repo types that need deterministic standardized tools and tasks.
- Tool creates tooling, tasks, and config files only. It does not create language/app skeletons like source files, packages, modules, or app templates.
- Config files come from base templates, then limited repo-specific tweaks may be applied by agent/user for ignores or local needs while staying grounded in copied template.
- After scaffolding, extension should queue/inject agent context listing created and touched files, then ask agent to inspect repo and propose recommended repo-specific tweaks such as ignore/include paths or local semantics.
- Post-scaffold agent work proposes changes to user; it does not silently mutate files as part of scaffold command.
- End of scaffold asks user whether to run post-scaffold repo-discovery recommendations now. It does not trigger automatically without confirmation.
- Post-scaffold discovery is proposal-only: inspect repo and present recommended tweaks, then stop. User decides any next action in normal chat.
- Do not add an always-available agent tool for scaffold template context because scaffold context is rarely needed outside setup/follow-up flows and should not occupy tool context.
- `/scaffold refine` triggers the same proposal-only repo-discovery/template-marker follow-up out of band after initial scaffold.
- `/scaffold refine` inspects repo structure and scaffold template markers/placeholders, then proposes repo-specific updates and stops.
- Use explicit refinement markers only in intentionally incomplete placeholder files, such as Xcode tasks. Do not add markers to normal complete generated configs.
- Templates use hybrid structure: static template files for config bodies, typed TypeScript manifest for profiles, tools, version sources, file targets, and tasks.
- Template files may use small explicit placeholder syntax for deterministic value replacement.
- Template renderer should stay minimal and parse/replace known placeholders only, not become a general expression language.
- Template syntax supports variables only, using `{{name}}`-style placeholders. No conditionals, loops, or filters.
- Templates are extension-owned only. Do not add custom project/global template discovery now.
- Command surface is single `/scaffold` command with optional profile args. No args opens full wizard flow.
- `/scaffold <profile...>` preselects requested profiles but still runs interactive version selection, conflict choices, and path/action preview.
- `/scaffold doctor` runs scaffold doctor/high-level alignment checks.
- Generated task implementation primarily lives in `.mise/tasks/` shell scripts.
- Shared shell helpers live under `.mise/tasks/lib/`.
- Shared shell helpers provide output formatting and command runner/capture/truncation helpers, not a full shell framework.
- `mise.toml` owns aggregate task definitions and tasks that do not make sense as file tasks.
- Do not mirror generated tasks into `package.json` scripts.
- Task scripts are designed for agent use and token-conscious output: attempt safe fixes first where appropriate, then report only remaining errors/warnings with tool-specific truncated output.
- Shared shell helpers may be scaffolded when they reduce repeated task logic across generated `.mise/tasks/` scripts.
- Extension should include a doctor/check command that reports whether repo scaffold structure aligns with expected generated structure at a high level. It should check presence/consistency of expected tools, tasks, and setup files, not perform line-by-line template validation.
- `/scaffold doctor` is advisory standards health inspection, not scaffold compliance validation.
- Doctor infers relevant standards from repo evidence and reports missing/broken validation wiring or suggested scaffold profiles. It does not compare against prior wizard choices because no scaffold state is persisted.
- Doctor detects scopes from repo evidence: TS from `package.json`, `tsconfig.json`, or JS/TS files; Markdown from Markdown files; YAML from YAML files; Go from `go.mod` or Go files; Swift from `Package.swift` or Swift files; Xcode from `.xcodeproj` or `.xcworkspace`.
- Doctor checks validation wiring: quiet-validator config exists in effective/project Crumbs config, detected configured scopes have expected `check:<scope>` entries, Xcode may be disabled placeholder, and configured quiet tasks exist in `mise tasks ls`.
- Doctor checks task/tool structure at high level: `.mise/tasks/check:<scope>` exists for configured scopes, referenced intent task files exist where inspectable, and expected tool pins exist in `mise.toml` for configured scopes.
- Doctor reports advisory statuses like `ok`, `missing`, `disabled`, and `suggested`; it does not run validation tasks, perform line-by-line template diff, or fail except for internal doctor errors.
- Extension should expose its built-in best-practice templates in an agent-discoverable way so agent can compare repo setup against current extension guidance when useful.
- Product intent is all-in-one standards driver for supported repo setup, not a deliberately narrow starter scaffold.
- Pushback during interview should focus on architecture, unnecessary complexity, and maintainability risks, not arbitrarily shrinking product scope.
- Interview questions should present only options grounded in verified docs/code/evidence. If viability is uncertain, inspect first instead of asking user to decide against speculative options.
- Every generated quality/check task follows an agent-optimized wrapper contract.
- Wrapper implementations can be thin direct commands when a tool already satisfies contract; do not force fake parsing logic.
- Each generated quality/check task should define fix strategy, error/warning extraction, truncation, and summary behavior.
- Formatter and linter tasks default to fix-first behavior when the selected tool supports safe autofix, then report remaining errors/warnings.
- Typecheck, test, and build tasks are verify-only.
- Compiled-language aggregate checks always include explicit build tasks in addition to tests when build tasks are configured.
- Fix-first behavior is part of standard generated formatter/linter behavior, not a per-task user configuration choice.
- Scaffolded validation tasks primarily exist as quiet-validator targets triggered by changed files, not as public commands agents run manually.
- Avoid broad public aggregate tasks like `format`, `lint`, `check`, and `build` unless they directly support quiet-validator or setup workflows.
- Scaffolding should configure `extensions.quietMiseTask.configs` so quiet validators run appropriate mise tasks for changed file scopes.
- This scaffold extension exists partly to make quiet validators robust and deterministic across repos.
- When validation tasks are scaffolded, automatically write/update quiet-validator config for selected profiles.
- Quiet-validator config writes should respect existing project Crumbs config when present; otherwise create standard project config at `.pi/crumbs.json`.
- Task taxonomy combines scoped intent tasks with scoped aggregate check tasks.
- Scoped intent tasks handle concrete behavior such as `lint:<scope>`, `format:<scope>`, `test:<scope>`, `typecheck:<scope>`, or `build:<scope>`.
- Scoped aggregate tasks like `check:<scope>` run relevant intent tasks for the same scope and serve as quiet-validator targets.
- Canonical task matrix:
  - `ts`: `format:ts`, `lint:ts`, `typecheck:ts`, `test:ts`, aggregate `check:ts`.
  - `markdown`: `format:markdown`, `lint:markdown`, aggregate `check:markdown`.
  - `yaml`: `format:yaml` when formatter available, `lint:yaml`, aggregate `check:yaml`.
  - `go`: `format:go`, `lint:go`, `test:go`, `build:go`, aggregate `check:go`.
  - `swift`: `format:swift`, `lint:swift`, `test:swift`, `build:swift` for SwiftPM, aggregate `check:swift`.
  - `xcode`: `build:xcode`, `test:xcode`, aggregate `check:xcode`.
- Canonical scopes are language/filetype-oriented: `ts`, `markdown`, `yaml`, `go`, `swift`, and `xcode`.
- Selected tools are implementation details behind stable scopes.
- `ts` scope covers JS/TS files together, including `.js`, `.jsx`, `.ts`, and `.tsx`.
- `swift` and `xcode` are separate scopes: `swift` owns Swift source style/package-oriented checks, while `xcode` owns Xcode project build/test checks.
- Xcode tasks may be scaffolded as editable placeholder task files because repositories may be scaffolded before an Xcode project/scheme exists.
- Xcode placeholder tasks should make required scheme/project/destination edits obvious for later agent/user modification.
- Xcode quiet-validator config is generated disabled until placeholder task details are filled and ready to enable.
- Quiet validators README should document per-config `enabled` support because parser already supports it and scaffold relies on it.
- TS testing uses Bun test when Bun is primary runner.
- TS testing uses Vitest when Node+npm is primary runner, including any needed Vitest scaffolding/config.
- TS scope always includes test task/framework setup.
- Generated test tasks pass with a concise `no tests found` summary when no tests exist.
- Go profile includes built-in Go tooling and `golangci-lint`: formatting, tests, build, vet, and lint.
- Swift profile includes SwiftFormat, SwiftLint, and built-in SwiftPM/Xcode build/test tasks where selected.
- YAML profile includes `yamllint`. YAML formatting is included only when selected formatter tooling supports YAML cleanly.
- Python runtime/tooling may be included as baseline support because agents commonly reach for Python while coding in repos.
- Markdown formatting should include `mdformat` via mise-managed Python/pipx tooling if selected, with exact pinned versions.
- Markdown formatter stack targets GitHub-ish Markdown using `mdformat` plus plugins: `mdformat-gfm`, `mdformat-frontmatter`, `mdformat-footnote`, and `mdformat-gfm-alerts`.
- mdformat plugins are handled as one bundled GitHub Markdown plugin-set selection. UI shows exact resolved versions for every plugin, and output pins each package exactly in mdformat install args.
- Do not include Markdown local relative-link checking or `markdownlint-rule-relative-links`; custom rule/module resolution is more hassle than desired.
- Include an `Agent Python baseline` scaffold item, default-on, because coding agents often use Python for advanced file writing/manipulation.
- Agent Python baseline pins exact Python version in `mise.toml`.
- Agent Python baseline pins exact `python` and `uv` versions in `mise.toml`.
- Python/uv selection may present combo choices for convenience, but final output always writes exact pins for both tools.
- Python-backed package version lists should filter to versions compatible with selected Python version when registry metadata exposes `requires_python`.
